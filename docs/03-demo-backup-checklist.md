# Demo Backup & Delivery Checklist

Run through this in the days before the session, and again the morning of.

## Recorded fallbacks (do this first — needs the most lead time)

- [ ] Record Tier 1 demo end-to-end (messy doc → structured Word/Excel/PDF output)
- [ ] Record Tier 2 demo end-to-end (RFP assessment agent, or inbox search → summarize → draft → follow-up doc)
- [ ] Record Tier 3 demo end-to-end (coding agent fixing a test / refactoring / new module)
- [ ] Store all three recordings somewhere reachable without internet (local drive + cloud backup)
- [ ] Time each recording — trim to fit inside its tier's window with room to talk over it

## Content prep

- [ ] Anonymize any real client data used in demo inputs (RFPs, requirement docs, procurement docs)
- [ ] Confirm the "messy" input docs are messy enough to be convincing but not so messy the demo stalls
- [ ] Pick and rehearse exactly one hands-on task for the 80–90 min segment — test it on a fresh account to confirm it works with zero prior setup
- [ ] Draft the 2–3 concrete next steps for the close (tool access to request, one workflow to try this week, where to ask for help)

## Tool & access checks (morning of)

- [ ] Confirm live access to whatever tool is used for Tier 1 (file generation) — test on the actual presentation machine, not just your laptop
- [ ] Confirm live access for Tier 2 (Gmail/Drive or equivalent connectors) — check auth hasn't expired
- [ ] Confirm live access for Tier 3 (coding agent + repo access) — check the target repo/branch is in the expected starting state
- [ ] Test the room's wifi/hotspot backup — have a phone hotspot ready as a second option
- [ ] Charge and test the presentation laptop; disable auto-updates/notifications

## Room logistics

- [ ] Identify and privately brief the audience "plant" for the hands-on segment — someone comfortable going first and narrating their result out loud
- [ ] Confirm screen sharing/projector resolution works with your demo tool's UI (some agent UIs are dense — test readability from the back row)
- [ ] Have the recorded fallbacks cued up and ready to switch to within 10 seconds if a live demo stalls
- [ ] Bring a physical printout or slide with the 2–3 close-out next steps, in case the last few minutes run short

## During the session

- [ ] If a live demo is slow, narrate while it runs rather than going silent — use the wait time to explain what's happening under the hood
- [ ] If a live demo fails twice, switch to the recording without apologizing extensively — say "let me show you the version I captured earlier" and move on
- [ ] Watch the clock at each tier boundary — better to cut a demo short than to compress the hands-on close
