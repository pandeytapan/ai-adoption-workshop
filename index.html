What "paste and copy" actually looks like

Pasted in"We need bulk invoice uploads, an approval step before finance sees it, and search that doesn't crawl once we're past 10k records. Also has to work with our SSO. Rough timeline?"
Copied out"Here's a summary: 1) Bulk upload 2) Approval workflow 3) Faster search 4) SSO integration. Let me know if you'd like more detail!"

→ then copied out and manually reformatted into whatever the client actually needs. That reformatting step is the 90%.

Set up the pain before naming it: "How many of you have taken a reply like this and manually turned it into something a client can actually use?" This is the exact shape of what most people mean when they say "I use AI for work" — and it's the ceiling we're about to blow past.

00 — Reframe, not features

Today vs. what's possible

TODAY

Paste → Copy

A paragraph in, a paragraph out. You do the reformatting, the cross-checking, the follow-up — by hand.

POSSIBLE

Read → Act → Flag

An agent reads a doc, checks three sources, drafts a real deliverable, and flags what still needs a human.

"If you're only using AI for snippets, you're using 10% of what's available and doing 90% of the work yourself."

Land this line and let it sit. Don't demo anything yet — the reframe needs to land on its own first. Frame the three tiers as a ladder: Tier 1 — it produces things. Tier 2 — it does things across systems. Tier 3 — it works inside your codebase like a teammate.

Tier 1 — Takeaway

What made the second prompt work

  • Named the deliverable and its format — ".docx", not "a summary" — removes guessing about output shape.
  • Specified the exact sections — exec summary, grouped requirements, open questions, timeline — instead of "organize it well."
  • Asked it to group and number, forcing the categorization the messy input never had.
  • Asked for open questions explicitly — surfaces uncertainty (SSO scope, mobile priority) instead of silently guessing an answer.

Vague prompt → vague shape. Specific deliverable → specific shape. The model mirrors the structure you hand it.

This is the bridge slide into "why" — keep it tight, then move to the technical explanation.

Tier 1 — Under the hood

Why the same model gives two different outputs

Not bag-of-words

Older NLP (bag-of-words / TF-IDF) treated text as an unordered set of words — order and structure were discarded. An LLM tokenizes into subword units but keeps every position, via positional encoding.

Self-attention

Every output token is computed by attending over all prior tokens. Your instruction words — "docx", "grouped", "open questions" — sit in the same attended context as the input doc, at every generation step.

That's why four extra format words in the prompt measurably reshape every sentence generated after them — the instruction isn't a separate channel, it's part of the same context the model attends over.

Keep this conceptual — attention weights, not the full transformer math. Save equations for a whiteboard follow-up if someone asks.

Tier 1 — Under the hood

Next-token prediction + the feedback loop

  • At its core, the model predicts one next token at a time, conditioned on everything before it — repeated thousands of times to produce a response.
  • Pretraining alone gives fluent, plausible completions — it doesn't reliably follow explicit instructions yet.
  • RLHF / instruction-tuning layers a feedback loop on top: human raters (or a reward model trained on human preferences) score candidate outputs, and the model is nudged — via gradient updates — toward completions that satisfy explicit asks like "produce sections X, Y, Z."
  • That's the mechanism behind today's demo: a clearer instruction gives the reward-shaped model a sharper target to predict toward, one token at a time.
This closes the technical arc: tokenization+attention explain "how it reads," RLHF explains "why it obeys." Both matter for why prompt quality changes output quality.

Tier 3 — Live demo

Reading the codebase before touching it

  • Show a real failing test or messy function first, plainly — so the room can judge the before/after themselves.
  • Ask it to fix, refactor, or scaffold a new module following existing project conventions.
  • Narrate what it's reading, not just what it's writing — it checks other files before it changes anything.

"That's not autocomplete. That's delegation."

If code review comes up: agent output still goes through the same PR review process as anyone's code. Nothing merges without a human.
